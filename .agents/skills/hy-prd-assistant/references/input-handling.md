# 輸入類型處理策略

不同輸入類型的資訊密度、結構與可信度不同。本文件定義各類型的**萃取策略**與**對應到 PRD 章節**的方式。

## 會議記錄

**特徵**：含討論脈絡、決策、待辦、分歧意見。內容跳躍，一段討論可能橫跨多個議題。

### 萃取步驟

1. **先掃一遍全文**，標記每一段落屬於哪一類：
   - `[背景]` — 為什麼要做、現況問題
   - `[需求]` — 功能描述、使用者需求
   - `[決策]` — 已經拍板的結論
   - `[待議]` — 尚未決議、有分歧
   - `[行動]` — 待辦事項、負責人
2. **萃取結構化資訊**：

   | 會議記錄內容 | 對應 PRD 章節 |
   |---|---|
   | `[背景]` | 背景簡介 → 專案背景 |
   | `[決策]` 中的目標/價值 | 背景簡介 → 核心價值 / 專案目標 |
   | `[需求]` | 使用者故事、功能清單 |
   | `[決策]` 中的流程/規則 | 功能流程規格 |
   | `[待議]` | 附錄 → 未決議項目 |
   | `[行動]` 中的 owner | 功能清單 → 負責人 |

3. **處理分歧**：若會議中出現「A 方案 vs B 方案」尚未決議，在對應功能標註：
   ```
   【待確認：規格選型】
   - 方案 A：<描述>
   - 方案 B：<描述>
   - 會議討論：<重點>
   ```

### 常見陷阱

- **不要把討論過程當結論**：「有人提到可能要加 X 功能」不等於「需要 X 功能」。只有明確決策或多數共識才收進功能清單
- **發言人不等於負責人**：會議中提到某人要做某事，不一定代表他是該功能的 owner，有疑慮時標 `【待確認：負責人】`

## 需求訪談紀錄

**特徵**：通常是 PM/UX 與使用者 1-on-1 的對談，含痛點、期望、現有流程。資訊密度高但主觀。

### 萃取步驟

1. **識別角色**：訪談對象是誰？（一般員工、主管、IT、HR……）這個角色對應到使用者故事的「作為 X」
2. **分離三類資訊**：
   - **現況描述**（「我現在都要……」）→ 專案背景 / 痛點
   - **痛點**（「最困擾的是……」）→ 專案目標要解決的問題
   - **期望**（「我希望能……」）→ 使用者故事、功能需求
3. **轉換為 User Story**：

   | 訪談原文 | User Story |
   |---|---|
   | 「我希望填請假時能自動帶出代理人」 | 作為員工，我想要系統自動建議職務代理人，以便加快填表速度 |
   | 「主管不在家還要在 Line 追人很麻煩」 | 作為主管，我想要在手機上直接核准申請，以便不在辦公室時也能處理 |

4. **萃取具體欄位規則**：訪談中若有「通常 X 是 Y」這類具體資訊，直接填入對應欄位的「處理邏輯」或「預設值」

### 常見陷阱

- **不要照搬原文**：訪談語氣較口語，直接寫進 PRD 會顯得不專業。轉換為規格語言（「應」「需」「格式為」）
- **樣本數不足時要標註**：單一訪談對象說的話，若沒有其他驗證，在對應需求標 `【待確認：需多方驗證】`

## 頁面連結

**特徵**：使用者給一個 URL（可能是既有網站或參考案例），要從中反推功能規格。

### 萃取步驟

1. **判斷是否可抓取**：
   - 公開網站：用 WebFetch 或 firecrawl-scrape 取得 HTML
   - Figma 連結：**不要**嘗試抓取（需登入），直接放在「連結」欄位
   - 內部系統：請使用者改提供截圖
2. **分析頁面結構**：
   - 導覽區有哪些連結？ → 功能清單主架構
   - 頁面主內容是什麼類型？（表單 / 列表 / 卡片 / 詳情頁）
   - 有哪些互動元件？（按鈕、下拉選單、彈窗）
3. **反推功能規格**：
   - 從 DOM/截圖識別的欄位 → 填入功能及欄位說明
   - 從按鈕文字反推動作 → 列入流程起點或驗收條件
4. **標註推斷項**：只要是從頁面「反推」而非明確告知的規則，加 `【推斷】` 標記，例：
   ```
   - 預設值：【推斷】選單第一個項目
   ```

### 常見陷阱

- **頁面外的邏輯看不到**：送出按鈕背後的 API、驗證規則、錯誤處理，在純頁面分析中往往看不到，需標 `【待確認：後端邏輯】`
- **不要把 UI 樣式當需求**：顏色、間距、字體等視覺細節交由設計稿處理，PRD 只描述「功能與邏輯」

## 頁面截圖

**特徵**：使用者給一張或多張畫面截圖，要對照截圖產出 PRD。

### 萃取步驟

1. **用 Read 工具讀取圖片**（Read 支援圖片檔）
2. **逐區塊分析**：
   - 由上到下識別每個區塊（Header、主內容、Footer）
   - 每個區塊的互動元素：輸入框、選擇器、按鈕
3. **建立「圖 N」編號**：每張截圖給一個編號，在「雛形」欄位引用
4. **填寫欄位說明**：
   - 從截圖可辨識的：欄位名稱、顯示格式、是否有紅色必填標記
   - 從截圖無法辨識的：預設值、驗證規則、處理邏輯 → 標 `【待確認】`
5. **若有多個狀態截圖**（預設態、填寫態、錯誤態、成功態），分別說明每個狀態的觸發條件

### 判斷必填的視覺線索

| 截圖特徵 | 推斷 |
|---|---|
| 欄位名稱前有紅色 `*` | 必填：是 |
| 欄位名稱後有「(選填)」 | 必填：否 |
| 送出按鈕 disabled 但欄位已填部分 | 未填的欄位是必填 |
| 錯誤文案「請輸入 X」 | X 為必填 |

沒有明確視覺線索時，標 `【待確認：是否必填】`。

### 常見陷阱

- **不要腦補看不到的頁面**：截圖只顯示 A 頁面時，不要擅自描述 B 頁面的規格
- **注意視窗斷點**：若只有桌面版截圖，手機版規格需標 `【待確認：RWD 行為】`

## 混合輸入

實務上使用者常常同時給多種資料（例：會議記錄 + 訪談 + 截圖）。處理原則：

1. **依來源分類處理**：先把每份資料獨立萃取成結構化資訊
2. **交叉比對**：
   - 不同來源對同一需求的描述是否一致？
   - 有衝突時以哪個為準？（通常**決策層級**：正式決策 > 主管訪談 > 一般訪談；**時間**：最新 > 舊版）
3. **明確記錄資料來源**：在附錄的「參考資料」列出所有輸入來源

範例：

```markdown
## 附錄 - 參考資料

- 2026/04/15 需求討論會議記錄
- 2026/04/18 HR 部門訪談紀錄（受訪者：林主任）
- 2026/04/20 Figma 雛形：https://figma.com/...
- 2026/04/21 既有 OA 系統截圖（附件 screenshot-01.png ~ screenshot-05.png）
```

## 共通原則

1. **寧缺勿濫**：看不到的不要編造。`【待確認：xxx】` 比編造一個合理答案更有價值
2. **保留原始術語**：使用者訪談中特別提到的業務術語（如「私公用車」「差勤單」）照原樣使用，不要「翻譯」成一般詞
3. **量化**：能給具體數字的地方盡量給（例：「不超過 10MB」比「大小限制」清楚）
4. **一致的標註風格**：
   - `【待確認：<項目>】` — 需與使用者確認
   - `【推斷】` — 從其他資訊推論，可能需要驗證
   - `TBD` — 未來版本再補（用於負責人、時程等）
