## 1. 封面

必要欄位：

- 文件名稱（例：XX 系統 PRD）
- 版本號（例：v0.1）
- 發布日期（YYYY/MM/DD）
- 作者 / 負責 PM
- 文件狀態（Draft / Review / Approved）

## 2. 修訂追蹤表

表格欄位：`版本 | 修訂日期 | 修訂人 | 修訂內容`

- 版本號採 `v主版本.次版本`（例：v0.1、v1.0、v1.1）
- 首次建立寫入 v0.1，後續每次修訂新增一列
- 修訂內容寫「做了什麼」，不寫「為什麼」

## 3. 背景簡介

分三個子章節，各以條列呈現：

### 3.1 專案背景

說明**現況痛點**或**驅動改變的原因**。撰寫提示：

- 為什麼現在要做這件事？
- 既有系統/流程有什麼問題？
- 組織環境有什麼變化？

避免只寫「我們要做一個 X 系統」。專案背景要說明「為什麼需要這個專案」。

### 3.2 核心價值

說明**這個專案帶來的價值**（對使用者、組織、技術面）。與專案目標的差異：

- 核心價值 = 長期、抽象的「好處」（例：降低維護成本、提升一致性）
- 專案目標 = 本次可量測的「產出」（例：遷移 10 個行政流程）

### 3.3 專案目標

本次專案**可驗收的目標**。盡量符合 SMART 原則（具體、可衡量、可達成、相關、有時限）。每項目標一行。

## 4. 使用者故事（User Stories）

**格式**：`作為 <角色>，我想要 <功能/行為>，以便 <目的/價值>`

**分組方式**：依角色分組（例：一般員工、主管、系統管理員、HR）

**撰寫要點**：

- 一個故事聚焦一個需求，不混多件事
- 目的/價值要寫出「為什麼」，不是重述功能
- 避免寫成實作細節（❌「作為員工，我想要點擊送出按鈕」）

**範例**：

```
### 一般員工
- 作為員工，我想要在手機上填寫請假申請，以便出差時也能及時送審
- 作為員工，我想要看到最近使用過的表單，以便不用每次都找目錄

### 主管
- 作為主管，我想要一次看到待審批清單，以便批次處理
```

## 5. 功能清單

表格欄位：`序號 | 主要功能 | 次要功能 | 功能描述 | 負責人 | 優先級`

| 欄位 | 說明 |
|---|---|
| 序號 | `1.1`、`1.2`、`2.1` 等階層式編號，對應到主/次要功能 |
| 主要功能 | 功能模組名稱（例：流程表單、差勤管理） |
| 次要功能 | 模組下的具體功能（例：填寫請假申請、查詢假單） |
| 功能描述 | 一句話說明這個功能做什麼（不寫實作細節） |
| 負責人 | PM / PO / Team Lead 姓名（可填 TBD） |
| 優先級 | P0 / P1 / P2，或 MUST / SHOULD / COULD |

**規則**：

- 每一列的「次要功能」都要在「功能流程規格」章節有對應的詳細表
- 優先級若不確定，標為 `【待確認】` 不要自行決定

## 6. 流程圖

獨立章節，呈現**跨頁面/跨角色**的主要流程。使用 Mermaid flowchart，細則見 [flow-diagram.md](flow-diagram.md)。

**何時放在此章節 vs. 功能流程規格內**：

- **本章節**：跨多個功能、涉及多角色的主流程（例：請假從申請到核准完整流程）
- **功能流程規格內**：單一功能的操作流程（例：填寫請假表單）

## 7. 功能流程規格

每個「次要功能」對應一張表。完整格式與範例見 [functional-spec-format.md](functional-spec-format.md)。

核心欄位：

- 連結（Figma / 設計稿）
- 流程說明
- 流程起點
- 適用對象與場景
- 流程圖（Mermaid）
- 雛形（畫面截圖引用）
- 功能及欄位說明
- API 規格
- 驗收條件（AC）
- 錯誤處理

## 8. 畫面雛形

若有 Figma 連結，列出各頁面的連結對應表：

```
| 頁面 | Figma 連結 |
|---|---|
| 流程表單目錄 | https://figma.com/... |
| 請假申請表單 | https://figma.com/... |
```

若使用者提供截圖，以圖片引用格式呈現並編號：

```markdown
圖1 流程表單目錄
![流程表單目錄](assets/screenshots/form-catalog.png)
```

圖片編號在功能流程規格的「雛形」欄位被引用（例：見圖1、圖2）。

## 9. 附錄

選填，適合放：

- **術語表**：專案內的縮寫或專有名詞
- **參考資料**：會議記錄連結、訪談紀錄連結、競品分析
- **未決議項目**：待下一版確認的問題清單

## 驗收條件（AC）撰寫格式

推薦兩種格式擇一，全篇保持一致：

### 格式 A：Given-When-Then

```
- Given 使用者已登入且為正職員工
  When 點擊「流程表單目錄」的「請假申請」
  Then 系統顯示請假申請表單，申請人欄位預設為本人
```

### 格式 B：條列式

```
- 必填欄位未填時，送出按鈕應為 disabled 狀態
- 送出成功後顯示綠色 Toast「申請成功！已進入審核流程」
- 送出失敗後顯示紅色 Toast「請檢查內容或稍後再試」
- Toast 於 10 秒後自動消失，亦可手動關閉
